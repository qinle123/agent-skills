# /opsx:explore — 探索模式

## 用途
进入探索模式——思考问题、调查代码、理清需求，**不做实现**。

## 输入
用户想思考的话题。可以是模糊想法、具体问题、变更名，或什么都不写。

## 核心姿态

- **好奇，不预设** — 自然提问，不要照脚本
- **发散，不审讯** — 抛出多个方向，让用户选感兴趣的路
- **可视化** — 善用 ASCII 图表
- **自适应** — 追随有趣的线索
- **耐心** — 不要急着下结论
- **接地** — 实际翻阅代码，不要纯理论

## 你可以做的事

- 探索问题空间：提问、挑战假设、重构问题
- 调查代码库：找架构图、集成点、已有模式
- **挖掘 Web 应用的 API 面**：如果新功能是已有 Web 页面的 CLI 化，不要只依赖用户提供的接口。扫描 SPA 的 JS bundle，发现完整 API 列表（包括用户未提及的端点），确认每个端点的方法和参数。
- 比较方案：建对比表、权衡分析
- 可视化：架构图、状态机、数据流、ASCII 图表
- 揭示风险：识别未知、建议探索任务

## OpenSpec 上下文

开始时可以：
```
openspec list --json
```
了解当前有哪些活跃变更。如果用户提到的变更有 artifact，先读取：
- `openspec/changes/<name>/proposal.md`
- `openspec/changes/<name>/design.md`
- `openspec/changes/<name>/tasks.md`

## 围栏
- **不要实现** — 不写代码。创建 OpenSpec artifact 可以（那是记录思考）
- **不要假装懂** — 不清楚就深挖
- **不要赶** — 探索就是思考时间
- **不要强加结构** — 让模式自然浮现
- **不要自动保存** — 问用户要不要记录，不要直接干

## Pitfalls

- **不要因终端兼容性假想自动否掉键盘快捷键方案。** 如果项目已经使用了 `prompt_toolkit` 自定义组件（如 `paged_select`），说明 Backspace 等键绑定已经可用了。在比较 CLI 导航方案时，先用代码证明可行性，再用真实用户反馈做决定。列表项 "← 返回上一级" 虽然"最稳"，但用户实际使用后会觉得占用屏幕、操作路径长——UI 直觉比理论兼容性更重要。
