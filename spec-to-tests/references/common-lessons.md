# Common Lessons

这份文件用于沉淀跨项目可复用的自动化测试经验。

更新规则：

- 只记录通用问题与通用修复思路
- 不记录项目私有 URL、账号、token、时间戳、临时数据
- 优先写“信号 -> 可能原因 -> 推荐处理”

## Source Of Truth

### XMind 不是业务规则权威

信号：

- 只有 XMind，没有需求文档或验收标准
- XMind 写了预期结果，但没有说明业务原因
- XMind 与需求、接口或现有实现出现口径差异

推荐处理：

- 只把 XMind 作为覆盖校准来源
- 只有 XMind 时，矩阵行标记 `source=xmind` 和 `assertionAuthority=needs-confirmation`
- 需求到达后做 coverage diff，再把确认过的断言改为 `confirmed`

### XMind 晚到时应使用两阶段矩阵

信号：

- 开发已经开始或完成，测试同学的 XMind 后续才提供
- 早期需要 AI 根据需求先写测试计划或自测范围

推荐处理：

- 第一阶段根据需求生成 `source=requirement` 的初版矩阵
- 第二阶段根据 XMind 生成候选矩阵
- 用 reconcile 脚本或人工审查输出 covered / missing / xmind-only / conflicts
- 只对最终矩阵生成测试代码

## Input Processing

### 大体积结构化输入不应直接塞进上下文

信号：

- 输入是 xmind、导出的复杂树结构、超长 checklist
- 直接读取原文件会浪费 token，且不利于后续重复执行

推荐处理：

- 优先用脚本把结构化文件转成 markdown 或 json
- 把解析逻辑沉淀到 `scripts/`，不要每次临时手写
- 后续步骤消费脚本输出，而不是反复读取原文件

### XMind 叶子节点经常拆得过细，不应直接平铺成 matrix

信号：

- XMind 里同一层连续出现大量“文件编号 / 文件名称 / 版本号 / 状态”之类叶子节点
- 自动生成的 matrix 看起来很完整，但几乎每条都只是静态字段或列名
- 后续很难判断哪些是真正需要自动化的业务用例

推荐处理：

- 先把脚本输出当成候选矩阵，不要直接作为最终 matrix
- 按“一个业务目标 / 一条业务链路 / 一组同类规则”合并叶子节点
- 把字段名、列表列、详情静态展示项收进父级用例的断言
- 只有独立交互、独立状态流转、独立校验规则或独立风险点，才保留单独用例

### 脚本命令应优先显式使用可用解释器

信号：

- 文档里的 `python` 命令在本机不可用
- 同一套脚本换台机器后直接执行失败

推荐处理：

- 命令示例优先写成 `python3`
- 或显式沿用项目当前解释器
- 不要默认假设 `python` 一定存在且指向 Python 3

## Selector And Interaction

### 稳定 `data-testid` 应作为自动化交互锚点的通用基准

信号：

- 需要和页面元素稳定交互
- 现有定位依赖按钮顺序、模糊文本、aria、样式类名或复杂 DOM 结构

推荐处理：

- 只要项目允许改前端代码，就优先补稳定 `data-testid`
- 页对象和测试默认优先使用这些稳定锚点，而不是猜测 DOM 结构
- 只有在明确不能补测试锚点时，才退回文本、role 或容器级定位
- 新增 testid 后，仍要验证它是否对应真实可点击元素，而不是只挂在外层容器上

### 下拉框在真实环境下不稳定

信号：

- 下拉已展开，但选项点击无效
- 使用 aria 或模糊文本时偶发失败

推荐处理：

- 如果项目允许改代码，优先给触发器和选项补稳定 `data-testid`
- 在可见 dropdown 容器中按选项文案点击
- 避免依赖不稳定 aria 属性
- 选择前后补等待，避开动画阶段

### Confirm 类弹窗不应只靠普通 dialog 文本按钮定位

信号：

- 页面使用 `Modal.confirm`、Confirm Dialog 或框架级确认弹窗
- 测试看得到弹窗，但点击“确定”经常失败、卡住或点不到真实主按钮

推荐处理：

- 先识别真实弹窗容器，再点击主确认按钮
- 点击后显式等待弹窗关闭和页面刷新完成
- 不要假设所有确认弹窗都能用同一套普通 `dialog -> 按钮文本` 方案处理

## Auth And Session

### 真实环境 e2e 优先支持 token 注入，避免人工登录循环

信号：

- 每次启动都跳登录页
- SSO、重定向或登录回跳导致自动化不稳定
- 需要人工登录才能继续跑用例

推荐处理：

- 优先支持通过环境变量注入真实 token
- 在页面初始化前写入前端实际读取的存储键
- 只有 token 注入不可行时，再退回人工登录或会话复用

## Test Stack And Layering

### 测试栈明显不稳定时应及时切换

信号：

- React hook、实例、portal 或环境适配问题反复出现
- 为了让测试跑通，不得不堆积大量与业务无关的兼容代码

推荐处理：

- 先判断问题是测试栈还是业务代码
- 如果是测试栈兼容性问题，及时切换到更稳的现有方案
- 优先选择项目上下文里摩擦最小的组合，而不是执着于某个默认框架

### 仓库声明有 Jest/umi test，不代表当前测试基座真的可用

信号：

- `package.json` 里有 `umi test` 或 Jest 相关依赖
- 实际执行时卡在 Babel、preset、runner 或框架版本不兼容
- 仓库里几乎没有现成测试文件，说明这套基座可能长期未被真实使用

推荐处理：

- 不要只看脚本名判断“已有测试栈可复用”
- 先实际跑一次最小测试入口确认基座是否可用
- 若框架封装层已坏，优先为当前目标补一份独立、最小的 Jest 配置
- 依赖版本要与仓库主栈对齐

### 重型 UI 外壳不应主导组件测试

信号：

- 失败集中在 modal、drawer、portal、动画、滚动锁定
- 业务表单本身没有问题，但组件测试频繁卡在 UI 容器行为

推荐处理：

- 在 component 测试里对重型外壳做最小 mock
- 保留 open、submit、字段回填、校验等业务关键路径
- 把 UI 框架自身行为留给上层 e2e 或框架本身保证

## Data And Isolation

### 用例之间存在隐式依赖

信号：

- 单跑失败，全跑成功
- 后续用例依赖前一条用例建好的数据

推荐处理：

- 为关键场景补按需建数逻辑
- 支持只跑当前失败用例
- 降低串行链路长度

### 自动生成测试全部平铺，后续维护成本高

信号：

- 所有 `spec/page/mock/fixture` 都堆在同一级目录
- 功能模块一多就难以定位和重构

推荐处理：

- 自动生成测试按功能模块分目录
- 一个模块内聚合该模块的 spec、page object、fixture、mock
- 只有项目已有更强约定时才不按模块拆目录

### 自动生成代码直接落盘，容易把错误目录结构写进仓库

信号：

- 需求还没确认，脚本已经生成大量测试文件
- 路径或模块名判断错误后，清理成本高

推荐处理：

- 骨架生成默认先 dry-run
- 先让用户或执行者确认目录与文件计划
- 确认后再显式 `--write`

### 覆盖矩阵未经用户检阅就继续生成代码，返工成本很高

信号：

- XMind 刚解析完就直接开始写测试
- 后续用户才指出分层不对、范围有变更、某些 case 不该自动化

推荐处理：

- 先展示 matrix 给用户检阅
- 等用户确认最终 matrix 后再生成代码
- 用户对 matrix 有调整时，先改 matrix，再继续后续步骤

## Contract Mismatch

### 前后端枚举口径不一致

信号：

- 选择某个分类后，列表展示成另一个分类
- 导出参数、表单值、返回值彼此冲突

推荐处理：

- 以真实后端返回的 `id/name` 映射为准
- 前端常量、mock 数据、断言值同步修正
